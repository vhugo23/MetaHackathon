Three logical FastAPI services (deployable together at this scale, but designed with clean boundaries â€” a real microservices decomposition, not a monolith pretending to be one).

## 1. Service Boundaries

| Service | Owns | Why separated |
|---|---|---|
| **Inventory API** | Devices, configs, BGP session state | Changes independently of alerting logic; network engineers' primary interface |
| **Registry/DNS API** | Service registry, DNS record sync | Highest-consistency-requirement service â€” isolating it limits blast radius of bugs elsewhere |
| **Ops API** | Alerts, remediation actions, incidents, postmortems | On-call/SRE-facing; different auth/RBAC profile than the other two |

## 2. Authentication

All services: API-key auth via `Authorization: Bearer <key>` header for machine clients (config-push automation, exporters); session-based auth for human users (dashboard/Slack-approval flows) with role claims (`viewer`, `operator`, `admin`) checked per-endpoint. See `09_SECURITY_ARCHITECTURE.md` for the full RBAC matrix.

## 3. Inventory API

| Method | Path | Description | Auth |
|---|---|---|---|
| GET | `/devices` | List all devices, filterable by role/vendor | viewer+ |
| GET | `/devices/{id}` | Device detail incl. last health snapshot | viewer+ |
| POST | `/devices/{id}/configs` | Submit new config (triggers schema validation) | operator+ |
| GET | `/devices/{id}/configs` | Config push history | viewer+ |
| POST | `/devices/{id}/configs/{config_id}/rollback` | Manually trigger rollback | operator+ |
| GET | `/bgp/sessions` | Current BGP session states, fleet-wide | viewer+ |
| GET | `/bgp/sessions/{id}/history` | State-change history for one session | viewer+ |

## 4. Registry/DNS API

| Method | Path | Description | Auth |
|---|---|---|---|
| POST | `/services/{name}/register` | Register a service instance | operator+ (or service-account key) |
| DELETE | `/services/{name}/instances/{ip}` | Deregister an instance | operator+ |
| GET | `/services/{name}/instances` | Healthy instances for a service | viewer+ (internal callers use this to bypass DNS if needed â€” a documented fallback path) |
| GET | `/dns/records` | Current DNS record set (mirrors CoreDNS zone state) | viewer+ |

## 5. Ops API

| Method | Path | Description | Auth |
|---|---|---|---|
| GET | `/alerts` | List alerts, filterable by severity/status | viewer+ |
| POST | `/alerts/{id}/acknowledge` | Ack an alert | operator+ |
| GET | `/alerts/{id}/rca` | LLM-generated RCA summary | viewer+ |
| POST | `/remediation/{alert_id}/execute` | Trigger a remediation action (auto or human-approved) | operator+ (admin required for routing-affecting actions) |
| GET | `/incidents` | List incidents | viewer+ |
| POST | `/incidents/{id}/postmortem` | Submit postmortem | operator+ |

## 6. Design Notes

- All list endpoints support `?since=` and `?limit=` â€” every dashboard query is time-boundable, a real operational requirement, not an afterthought.
- `GET /services/{name}/instances` is deliberately exposed as a **documented DNS-bypass fallback** â€” if DNS itself is down, a caller (or a client-side circuit breaker) can hit this endpoint directly, illustrating defense-in-depth for service discovery.
- Full OpenAPI 3.0 spec is auto-generated by FastAPI at `/docs` and `/openapi.json` on each service â€” kept in sync with code by construction, not hand-maintained separately.
